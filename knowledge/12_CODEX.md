# 💀 POWER RULE

ไม่มีพลังใด “ไร้ข้อแลกเปลี่ยน”

- พลังสูง → ต้องมีข้อเสีย
- สกิลโกง → ต้องมีข้อจำกัด
- Devil Fruit → ต้องมี counter

---

# ⚖️ BALANCE AXES

## 1. Power vs Cost
- ยิ่งเก่ง → ยิ่งมีต้นทุนสูง

## 2. Strength vs Control
- พลังมาก ≠ ใช้ได้เสถียร

## 3. Ability vs Environment
- ไม่มีพลังที่ไร้จุดอ่อนในทุกสภาพแวดล้อม

---

# 🧠 HAKI BALANCE RULE
- Haki ไม่ใช่ “ชนะทุกอย่าง”
- ใช้เพื่อ:
  - อ่านสถานการณ์
  - ป้องกัน
  - เพิ่มโอกาส ไม่ใช่การการันตีชัยชนะ

---

# 💀 DEVIL FRUIT BALANCE RULE
- ทุกผลต้องมี:
  - Weakness
  - Counter
  - Situation where it fails

---

# 🏴‍☠️ FACTION BALANCE RULE
- ไม่มี faction ที่ครองโลก 100% โดยไม่มีแรงต้าน
- ทุกอำนาจต้องมี “จุดเปราะบาง”

---

# 🧠 FINAL PRINCIPLE

> “ไม่มีอะไรในโลกนี้ที่ไร้จุดอ่อน”

# 🧭 1. PURPOSE OF THIS FILE (หน้าที่ของ Codex)

Codex ไม่ใช่ไฟล์เพิ่มเนื้อหาโลก

Codex ทำหน้าที่:
- ล็อก “ความจริงร่วม” ของทุกไฟล์
- ป้องกันข้อมูลขัดกันระหว่างระบบ
- เป็น reference หลักเวลาทุกระบบตัดสินเหตุการณ์
- เป็น “ศาลสูงสุดของจักรวาลเกม”

📌 Codex ไม่สร้างเกาะ / ไม่สร้าง NPC / ไม่สร้างเนื้อเรื่องใหม่

---

# ⚖️ 2. PRIORITY RULE (ลำดับความถูกต้องของระบบ)

เวลาข้อมูลขัดกัน ให้ใช้ลำดับนี้:

1. CODEx.md (สูงสุด)
2. SYSTEM.md
3. WORLD.md
4. FACTION.md / DEVIL_FRUIT.md
5. NPCS.md / ITEM.md / SHIPS.md / CREW.md
6. LOGS.md (ต่ำสุด / เป็นผลลัพธ์ ไม่ใช่กฎ)

---

# 🌍 3. GLOBAL CONSISTENCY RULE

## 3.1 โลกต้อง “ต่อเนื่อง”
- เหตุการณ์ต้องส่งผลต่ออนาคต
- ไม่มีการรีเซ็ตโลก
- ไม่มีการย้อนเหตุการณ์โดยไม่มีเหตุผลเชิงระบบ

## 3.2 ความจริงของโลก
- ทุกเกาะ / ทุก NPC / ทุก faction มี “ความเป็นอิสระ”
- โลกดำเนินต่อแม้ไม่มีผู้เล่น
- การกระทำทุกอย่างต้องมี “ผลกระทบ”

---

# 🧩 4. CROSS-SYSTEM VALIDATION

Codex ใช้ตรวจความถูกต้องของระบบอื่น

## ตรวจสิ่งต่อไปนี้:
- ค่า bounty ต้องสัมพันธ์กับความสามารถ / ชื่อเสียง
- Faction ต้องไม่ขัด lore โลก
- Devil Fruit ต้องไม่ซ้ำกฎการมี 1 ผู้ใช้
- NPC ต้องมีเหตุผลในการอยู่ในโลก
- เกาะต้องมี logic เศรษฐกิจ / สังคม / การปกครอง

---

# ⚔️ 5. COMBAT AUTHORITY RULE

ถ้ามีความขัดแย้งในการต่อสู้:

ให้ Codex ตัดสินว่า:
- ผลแพ้ชนะต้อง “มีเหตุผล”
- ห้ามชนะ/แพ้แบบไร้ตรรกะ
- สภาพแวดล้อม + กลยุทธ์ มีน้ำหนักเท่าพลัง

---

# 🏴‍☠️ 6. FACTION REALITY LOCK

- ทุก faction ต้อง “มีอยู่จริงในโลก”
- ไม่มี faction ที่ไม่มีผลต่อโลกโดยรวม
- faction ต้องมี:
  - เป้าหมาย
  - การกระทำ
  - ผลกระทบต่อโลก

---

# 💀 7. DEVIL FRUIT REALITY LOCK

- 1 ผล = 1 ผู้ใช้เท่านั้น (global unique rule)
- การเกิดใหม่ของผลต้องเกิดในโลกจริง (ไม่ใช่ระบบ abstract)
- ไม่มีผลที่ “ลบกฎโลกทั้งหมด” ได้
- Awakening ต้องมีต้นทุนหรือข้อแลกเปลี่ยน

---

# 🧭 8. NARRATIVE CONTINUITY RULE

- เหตุการณ์ทุกอย่างต้องสามารถ “ย้อนตรวจสอบได้”
- ไม่มี retcon ยกเว้น Codex update
- NPC ที่ตาย = ตายจริงใน timeline
- โลกต้องมี memory ของเหตุการณ์

---

# 🔗 9. SYSTEM LINK RULE

ไฟล์ทั้งหมดต้อง “อ้าง Codex โดยอ้อม”

- SYSTEM = logic engine
- WORLD = geography + lore
- FACTION = power structure
- NPCS = behavior simulation
- LOGS = history output

📌 Codex ไม่ถูกอ้างตรงทุกบรรทัด แต่เป็น “พื้นฐานทุกบรรทัด”

---

# 🧠 10. FINAL PRINCIPLE

> Codex ไม่ใช่เนื้อหา  
> Codex คือ “กฎที่ทำให้ทุกเนื้อหาอยู่ร่วมกันได้”

# 🧠 ENGINE_VALIDATOR_SYSTEM.md
## ONE PIECE RPG ENGINE — SYSTEM INTEGRITY CHECKER

---

# 🎯 PURPOSE

ระบบนี้ใช้ตรวจสอบว่า:
- ข้อมูลในไฟล์ทั้งหมด “ยังสอดคล้องกันหรือไม่”
- ไม่มี logic ขัดแย้งระหว่างระบบ
- โลกยัง “สมจริงตาม Codex”

---

# ⚙️ VALIDATION RULES

## 1. WORLD CONSISTENCY CHECK
ตรวจว่า:
- เกาะมีเหตุผลในการดำรงอยู่
- สังคม / เศรษฐกิจ / อำนาจ มี logic
- ไม่มีเกาะที่ “ไร้บทบาทในโลก”

---

## 2. FACTION CHECK
- ทุก faction ต้องมีผลกระทบต่อโลก
- ต้องมีศัตรู / เป้าหมาย / การกระทำจริง
- ห้าม faction เป็นเพียงชื่อ

---

## 3. NPC VALIDATION
- NPC ต้องมี:
  - เป้าหมาย
  - บทบาท
  - ความสัมพันธ์กับโลก
- NPC ห้าม “ลอยอยู่โดยไม่มีเหตุผล”

---

## 4. DEVIL FRUIT CHECK
- 1 ผล = 1 ผู้ใช้ (ต้องไม่ซ้ำ)
- ผลต้องมี logic การใช้งานจริง
- ห้ามมีพลังที่ override Codex

---

## 5. ECONOMY CHECK (ถ้ามีระบบเศรษฐกิจ)
- เงิน / bounty ต้องสัมพันธ์กับโลก
- รายได้ / ทรัพยากร ต้องมีแหล่งที่มา

---

# 🚨 ERROR TYPES

- LORE_BREAK → โลกขัด logic ตัวเอง
- POWER_IMBALANCE → ตัวละครโกงเกินระบบ
- FACTION_VOID → faction ไม่มีผลต่อโลก
- NPC_NULL → NPC ไม่มีเหตุผลในการมีอยู่

---

# 🧠 RESULT

- PASS = โลกสอดคล้อง
- WARN = มีความเสี่ยง
- FAIL = ต้องแก้ก่อนเล่นต่อ

# 🌍 WORLD_SIMULATION_LOOP.md
## ONE PIECE RPG ENGINE — AUTONOMOUS WORLD SYSTEM

---

# 🎯 PURPOSE
ทำให้โลก:
- เดินต่อเอง
- สร้างเหตุการณ์เอง
- เปลี่ยนแปลงเอง

---

# 🔁 SIMULATION CYCLE

โลกจะอัปเดตเป็นรอบ (Loop):

## 1. POLITICAL PHASE
- Faction เคลื่อนไหว
- รัฐบาลโลกออกคำสั่ง
- กองกำลังเปลี่ยนตำแหน่ง

---

## 2. PIRATE PHASE
- กลุ่มโจรสลัดเคลื่อนที่
- ล่า / หนี / ตั้งฐาน
- เพิ่ม bounty / ชื่อเสียง

---

## 3. ISLAND PHASE
- เกาะเปลี่ยนสถานะ:
  - สงบ
  - สงคราม
  - ควบคุมโดย faction

---

## 4. NPC PHASE
- NPC เปลี่ยนเป้าหมาย
- เกิดความขัดแย้ง / ความสัมพันธ์ใหม่

---

## 5. EVENT GENERATION
- เกิดเหตุการณ์ใหม่:
  - สงคราม
  - การลอบสังหาร
  - การค้นพบ Devil Fruit
  - การล่มสลายของเกาะ

---

# 🧠 RULE OF CHANGE

- ทุกการเปลี่ยนแปลงต้องมี “เหตุผล”
- ไม่มี event เกิดแบบไร้ที่มา
- โลกต้อง “จำอดีตของตัวเองได้”

---

# ⚙️ OUTPUT

ทุก loop จะสร้าง:
- World Update Log
- Faction Movement
- NPC Activity
- Island Status Change
